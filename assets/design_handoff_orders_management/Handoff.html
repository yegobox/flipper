3 · Mental model

Two independent axes, both scoped to the current store/branch:

  • Direction — Incoming: requests other store locations have sent to this store, asking this store to send stock. Outgoing: requests this store has sent to other locations.
  • Status — Pending: awaiting a decision. Approved: resolved, read-only history.

Each request groups one or more line items (product + quantity) moving from one location (from) to another (to). Incoming pending requests are actioned by this store (Produce / Approve / Void). Outgoing pending requests were created by this store and can still have their requested quantities edited while they wait on the receiving store.

5 · Screen anatomy

  • Header — title "Orders Management" + subtitle, right-aligned Pending/Approved segmented control (.seg).
  • Direction control — full-width Incoming/Outgoing segmented control (.seg.seg-lg) directly below the header, each option with an icon.
  • Stat card — shown only on the Pending tab: icon + count of pending requests for the active direction + "Pending Requests" label. Deliberately minimal — no secondary badge/decoration.
  • Section list — heading ("Received Orders" for Incoming / "Sent Orders" for Outgoing) + a vertical stack of order cards (.panel.sgroup, one per request), or an empty state if none match the filters.

Confirm before build 12 · Open questions for product/backend

  • What does "Produce" actually do? No backend action was specified; this reference renders it as a ghost button with no wired behavior.
  • Is "Void" ever enabled? Currently rendered permanently disabled — confirm whether there's an eligibility condition or it should be removed.
  • Per-item partial approval — can an incoming request be approved with a different quantity than requested per line, or is Approve all-or-nothing? The "Approved: n/n" display assumes partial tracking exists.
  • Outgoing self-edit — can the requesting store edit quantities after the receiving store has started acting on the request? What happens to an edit mid-flight?
  • Rejected/voided history — is there a third status the Approved tab should also filter out, or does a voided request disappear entirely?
  • Multi-location scale — does this list need search/pagination once a store has many transfer partners, or is the current unpaginated list sufficient?
