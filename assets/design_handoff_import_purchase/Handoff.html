1 · About these files

The files in src/ are design references built in HTML/CSS + React (via in-browser Babel). They are prototypes that demonstrate the intended look, layout, and behavior — not production code to ship as-is.

Your task is to recreate these designs inside the target codebase using its established framework, component library, state management, and conventions. If no front-end environment exists yet, pick the most appropriate framework for the project and implement there. Treat the HTML/CSS as the source of truth for visual spec and interaction; treat the React structure as a reference for component breakdown, not a required architecture.

Fidelity: high. Colors, type, spacing, radii, and interactions are final. Recreate them precisely using your codebase's primitives (e.g. map the tokens below to your theme).

Read first 2 · Brief for AI coding assistants

This handoff is self-sufficient. Work efficiently and do not burn tokens re-deriving what is already specified here.

  • Don't echo the spec. Don't re-read or paste large file contents back into chat. Reference token names and section numbers instead of reproducing values.
  • Read only what you need. The HTML/CSS in src/ is the source of truth. Open ipm.css + record-purchase.css for exact values; skim the .jsx only for component structure. Avoid loading every file at once.
  • Map, don't reinvent. Translate the tokens in §3 to your existing theme variables once, then reference them. Reuse existing components (buttons, inputs, modals, badges) rather than rebuilding from these class names.
  • Plan before editing. Produce a short component list and a token map, confirm with the user, then implement. Don't open a long trial-and-error loop.
  • No narration. Skip step-by-step play-by-play and verbose summaries. Make the change, state what changed in 1–2 lines.
  • Ask only blocking questions. If a detail isn't here (e.g. real API shape), ask one concise question rather than guessing across multiple turns.
  • Keep diffs tight. Edit in place with minimal surrounding context; don't rewrite whole files to change a few lines.

5 · Top bar

Sticky single row (.subbar, position: sticky; top: 0; z-index: 25) on white with a bottom hairline. Left → right:

  • Date filter — label "Import from / Purchase from" (changes with mode) + native <input type="date"> (38px tall).
  • Export — ghost button (icon + label), download glyph. Collapses to icon-only ≤880px.
  • Record Purchase — primary blue button (plus-doc icon + label). Opens the modal (§8). Collapses to icon-only ≤880px.
  • Spacer, then the mode segmented control on the right.

Segmented control (Import / Purchase)

Pill track --surface-3, 4px padding; an absolutely-positioned white thumb (.thumb, --shadow-sm) slides under the active tab via measured offsetLeft/offsetWidth with transition: left .22s cubic-bezier(.3,.7,.4,1). Active label color --accent-strong. Each tab has a 17px leading icon. Full-width (tabs flex:1) on mobile.

6 · Import mode

Purpose: review imported items, edit name/prices/variant, assign variants, and approve/reject (single or bulk).

Edit toolbar (desktop) — .editbar

  • Panel card. Top hint line: idle shows "Select a row below to edit…" (pencil icon, muted); when a row is selected shows "Editing #N {item}" + a Clear link.
  • Fields grid (repeat(auto-fit, minmax(165px,1fr))): Item name, Supply price, Retail price, Variant (searchable combo, §8-style).
  • Bottom action row: Save Changes (primary), Accept All (green), and the Status filter select pushed to the right.

Data table (desktop) — .tbl.import

CSS-grid rows. Columns: No · Item Name (+variant tag) · HS Code · Quantity · Retail · Supply · Status · Supplier · Date · Actions. Secondary columns (HS Code / Supplier / Date) are hidden when the "show secondary" tweak is off — grid template switches.

  • Row hover → --surface-2; selected row → --accent-wash bg + 3px left accent bar (inset 3px 0 0 var(--accent)). Click a row to load it into the edit toolbar.
  • Actions: round 38px Accept (green-wash→green on hover) and Reject (red-wash→red) icon buttons. stopPropagation so they don't trigger row-select.

7 · Purchase mode

Purpose: review recorded purchase invoices grouped by supplier, expand to see line items, assign variants, and approve/decline whole invoices.

Toolbar

  • Status filter select (default "Waiting"), spacer, and a pager ("1–4 of N" + prev/next arrows). Page size = 4 groups.

Supplier group — .sgroup

  • Panel card. Header (click to expand/collapse): "Supplier: {name} (count)" + "Invoice: {no}"; right side has a green time pill, an accent total pill (UGX, sum of supply×qty), Accept All (green-soft) / Decline All (red-soft), and a chevron button that rotates 180° when open.
  • Body (--surface-2): line-items table — No · Name (+variant) · Qty · Supply · Retail · Status. If an item has no variant, its status cell shows an "Assign variant" affordance (tag icon). Click any line → opens Assign Variant modal (§8).

Mobile (≤880px)

  • Header reflows: id + chevron on row 1; time/total pills wrap to a second line; Accept All / Decline All become full-width buttons.
  • Line items become tappable mini-cards (.licard) with a 3-col Qty/Supply/Retail grid and an "assign variant" hint.

8 · Record Purchase modal

Opens as an overlay on the live screen (not a separate page). Markup is namespaced under a .rp root so its generic class names (.modal, .scrim, .field, .btn) never collide with the host screen's styles — keep this namespacing when porting, or scope via your own component styles.

Critical: the scrim and modal must be opaque immediately. Do not drive their visible state purely from an entrance animation with opacity + fill-mode — if the renderer pauses the animation at frame 0 the modal becomes see-through. Animate transform only; keep base opacity: 1. (This was a real bug we fixed.)
  • Scrim: rgba(20,28,46,.5) + backdrop-filter: blur(2px), z-index 100. Click to close.
  • Modal: max-width 1040px, white, --radius-lg, --shadow-modal, z-index 101.
  • Header: accent-wash icon badge + "Record Purchase" + subtitle + close ✕.
  • Header fields: Supplier (2fr) + Supplier TIN (optional); then Invoice No · Purchase date · Payment type (3-col).
  • Line items: "Add from catalog" (searchable popover of priced items by tax category) and "New item". Empty state with box icon + two pills. Table columns: Item · Tax category · Qty · Unit price · VAT 18% · Amount · delete.
  • Tax logic: categories A (Exempt), B (18%), C (Zero), D (Deemed). VAT = net × 0.18 only for B. Totals block: Taxable (B), VAT 18%, Exempt/zero (A+C+D), and grand Total (UGX).
  • Footer: Cancel (ghost) · Save as Waiting (outline) · Save & Approve (primary, disabled until supplier + ≥1 item).
  • Close on: ✕, Cancel, scrim click, Esc, or after a save.

Mobile (≤720px)

  • Becomes a bottom sheet: full width, rounded top only, slides up (transform: translateY(100%) → 0).
  • Header fields stack to 1 column; line items become labelled cards (delete button top-right); totals stack with grand total full-width; footer buttons stack full-width (Save & Approve on top).

Searchable variant/catalog combo

Button-styled trigger opens a popover with a search input (auto-focus) + filtered option list; selected option gets a check; closes on outside-click. Reused for the Variant picker in Import mode.

10 · State & data

App-level

  • mode: "import" | "purchase" (segmented control)
  • date, density, accent, showSecondary (tweaks)
  • recordOpen: boolean → mounts the modal
  • items (import list), groups (purchase invoices)
  • importFilter (default "all"), purchaseFilter (default "waiting")

Entities (replace with real API shapes)

EntityFields
Import itemid, item, hs, qty, retail, supply, status(wait|approved|rejected), supplier, date, variant
Purchase groupid, supplier, invoice, time, status(waiting|approved|rejected), items[]
Line itemid, name, qty, supply, retail, status, variant
Record draftsupplier, tin, invoice, date, payment, items[{name, cat(A–D), qty, price}]

Key interactions

  • Accept/Reject item → flips status; Accept All approves all "wait" items in the current filter.
  • Group Accept All / Decline All → sets group + all its items' status.
  • Toasts appear bottom-center on actions (ink bg, ~2.6s, slide-up).
  • Saving a purchase fires an onSaved callback (wire to your create-purchase API), then closes.
