3 · Mental model & data flow

This screen does not create data by hand — it reviews data pulled from RRA. Two independent tabs, both scoped to a branchId:

  • Import — individual imported items (RRA selectImportItems). Each is reviewed, optionally priced/renamed, then approved (as a new variant or merged into an existing one) or rejected.
  • Purchase — supplier invoices (RRA trnsPurchase), each with line-item variants. Each line is mapped to an existing variant or left to create new; then the whole invoice is approved/rejected.
  1. Sync — POST /imports/sync or /purchases/sync pulls new records from RRA (cursor-based; advances a server-side date cursor). Returns a job.
  2. List — GET /imports / /purchases (filter by status) renders the table/cards.
  3. Review & map — user assigns variants (import: new/merge; purchase: per-line itemMapper).
  4. Approve / reject — POST returns a job; poll until done; refresh the list.

6 · Async job pattern (critical)

Sync, approve, and reject are not synchronous. The prototype models this with a job(label, commit, okMsg) helper in ipm-app.jsx; in production replace the timer with a real poll loop:

  • On action: set the affected row(s)/invoice to processing immediately and show a spinner toast (Approving…).
  • POST → get jobId → poll GET /…/jobs/:jobId (e.g. every ~1s).
  • On success: apply the final status, toast success, refresh the list. On failed: revert to pending, toast error with job.error.
  • While processing, hide/disable the action buttons and keep the item visible (don't let a status-filter drop it mid-job — only the group/row's committed status changes when the job resolves).
A purchase invoice stays under the Pending filter during processing (its committed status is still pending); only its line items show the processing badge. It leaves the list when the job commits.

7 · Top bar & sync

Sticky single row (.subbar): left → right —

  • Date filter ("Import/Purchase from" + <input type=date>) — display cursor hint; the server owns the real sync cursor.
  • Synced status — green dot + "Synced {relative}"; flips to spinner + "Syncing…" during a sync job.
  • Export (ghost), Add manually (ghost, Purchase only — see §10), Sync from RRA (primary; spinner + disabled while busy).
  • Mode segmented control (Import / Purchase) far right; full-width on mobile; Export/Sync collapse to icons ≤880px.

8 · Import mode

Review imported items; edit name/prices/variant; approve (new or merge) / reject; bulk Accept All.

  • Edit toolbar (desktop) — click a row to load it into Item name / Supply / Retail / Variant fields; Save Changes; Accept All; inline status filter. Mobile: edits open in a bottom sheet; a sticky bar holds the filter + Accept All.
  • Table → cards. Columns: No · Item · HS Code · Qty · Retail · Supply · Status · Supplier · Date · Actions (secondary columns hidden by a tweak). Each row: round Approve / Reject icon buttons. Mobile card shows Barcode, Qty, Supply, Retail, Supplier, Date.
  • Approve = New vs Merge. The Approve action opens a choice dialog (.choice radio cards): Create new variant (no target) or Merge into existing variant (searchable Combo → targetVariantId). Confirm → processing → approved.
  • Accept All approves every pending item in the current filter (one job).

9 · Purchase mode

Supplier invoices grouped into expandable panels; per-line variant mapping; approve/reject the whole invoice.

  • Toolbar: status filter (default Pending) + pager (4 invoices/page).
  • Invoice header: "{supplier} (n)" + "Invoice {no} · TIN {tin}"; right: "N need mapping" amber chip (count of !assigned lines), time pill, total pill (UGX), Approve / Decline. While a job runs the actions become a Processing badge.
  • Expanded body: info strip (Sales date · Payment · Tax category · Items), then line items (table desktop / mini-cards mobile): No · Name(+variant) · Qty · Supply · Retail · Mapping.
  • Line mapping (itemMapper) — tap a line to open the map dialog: Create new variant or Map to existing variant (Combo) + editable name/supply/retail. Sets the line assigned.
  • Approve builds itemMapper = { "<targetVariantId>": ["<purchaseVariantId>", …] } from lines mapped to existing variants; unmapped lines create new. Empty {} = all new. Toast reports "X merged, Y new".

10 · Manual entry modal (Record Purchase)

Out of connector scope. The RRA connector only ingests synced purchases — it has no create-purchase endpoint. This modal is an optional manual path; wire it to your own purchase-create API or drop it. It is namespaced under .rp and styled in record-purchase.css.
  • Overlay on the live screen (solid panel + dimmed scrim; animate transform only, never opacity-from-0). Header fields: Supplier (searchable + inline-create nested dialog with name/TIN/phone), Invoice No, Purchase date, Payment type.
  • Line items: add from catalog (priced, by tax category) or new; tax categories A/B/C/D, VAT = net×0.18 for B only; totals = Taxable B / VAT / Exempt(A+C+D) / grand Total.
  • Footer: Cancel · Save as Waiting · Save & Approve (disabled until supplier + ≥1 item). Closes on ✕/Cancel/scrim/Esc. Mobile = bottom sheet.
